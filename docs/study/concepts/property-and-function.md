# 프로퍼티와 함수의 역할

## 핵심 질문

1. 왜 객체 리터럴(프로퍼티)만 쓰지 않고 함수를 사용하는가?
2. 프로퍼티가 먼저인가, 함수가 먼저인가?
3. 클로저는 어떻게 탄생했는가?

---

## 역사적 흐름

```
+--------------------------------------------------+
|                  역사적 순서                       |
+--------------------------------------------------+
|  1. 함수 (1930년대)                               |
|     └─ 람다 계산법 (Alonzo Church)                |
|                                                   |
|  2. 클로저 (1970년대)                             |
|     └─ Scheme 언어에서 정립                       |
|                                                   |
|  3. 객체/프로퍼티 (1980년대)                       |
|     └─ OOP, Self 언어                            |
|                                                   |
|  4. JavaScript (1995년)                          |
|     └─ Scheme(함수/클로저) + Self(프로토타입) 결합 |
+--------------------------------------------------+
```

**결론:** 함수 → 클로저 → 프로퍼티 순서이다.

---

## 왜 함수를 사용하는가?

### 객체 리터럴만 사용할 경우

```js
var element = {
    key: "",
    value: ""
};

// 문제점
element.key = "아무나 수정 가능";  // ❌ 보호 불가
```

### 함수(클로저)를 사용할 경우

```js
var element = (function() {
    var _key = "";  // 보호됨

    return {
        getKey: function() { return _key; },
        setKey: function(k) { _key = k; }
    };
})();

// 장점
element._key;  // ❌ undefined (접근 불가)
```

## 프로퍼티 vs 함수의 역할

```
+--------------------------------------------------+
|              역할 구분                             |
+--------------------------------------------------+
|  프로퍼티 (Property)                              |
|  +-- 데이터 저장 (상태)                           |
|  +-- 정적 구조                                   |
|  +-- "무엇을" 담는가                              |
|                                                   |
|  함수 (Function)                                 |
|  +-- 행위 정의 (동작)                            |
|  +-- 동적 생성                                   |
|  +-- "어떻게" 처리하는가                          |
|                                                   |
|  클로저 (Closure)                                |
|  +-- 함수 + 환경(스코프) 기억                     |
|  +-- 비공개 상태 가능                            |
|  +-- 프로퍼티 보호                               |
+--------------------------------------------------+
```

## Heap과 Stack 관점에서 이해

### 메모리 구조 기본

```
+--------------------------------------------------+
|              메모리 구조                           |
+--------------------------------------------------+
|                                                   |
|  Stack (스택)                                     |
|  +-- 원시값 저장 (number, string, boolean)        |
|  +-- 함수 호출 정보 (Call Stack)                  |
|  +-- 참조 주소 저장 (Heap을 가리킴)                |
|  +-- LIFO (Last In First Out)                    |
|  +-- 함수 종료 시 자동 해제                        |
|                                                   |
|  Heap (힙)                                        |
|  +-- 객체 저장 (Object, Array, Function)          |
|  +-- 동적 메모리 할당                             |
|  +-- 참조로 접근                                  |
|  +-- GC가 관리                                    |
|                                                   |
+--------------------------------------------------+
```

### 객체 리터럴만 사용할 경우

```
+-------------------------+-------------------------+
|         Stack           |          Heap           |
+-------------------------+-------------------------+
|                         |                         |
|  element: 0xE001 ------>|  Object (0xE001)        |
|                         |  +-- key: "노출됨"       |
|                         |  +-- value: "노출됨"     |
|                         |                         |
+-------------------------+-------------------------+
|  특징:                                            |
|  - Stack에 참조 주소만 저장                        |
|  - Heap의 객체에 직접 접근 가능                    |
|  - 보호 메커니즘 없음                              |
+--------------------------------------------------+
```

### 함수 + 클로저 사용할 경우

```
+-------------------------+-------------------------+
|         Stack           |          Heap           |
+-------------------------+-------------------------+
|                         |                         |
|  [IIFE 실행 중]         |  익명함수 (0xF001)       |
|  _key: "" (지역변수)    |                         |
|  _value: "" (지역변수)  |                         |
|                         |                         |
+-------------------------+-------------------------+
|                         |                         |
|  [IIFE 실행 후]         |  Closure Scope (0xC001) |
|  element: 0xE002 ------>|  +-- _key: "보호됨"      |
|                         |  +-- _value: "보호됨"    |
|                         |       ^                 |
|                         |       | 참조            |
|                         |  Object (0xE002)        |
|                         |  +-- getKey: 0xG001 ----+
|                         |  +-- setKey: 0xS001 ----+
|                         |                         |
+-------------------------+-------------------------+
|  특징:                                            |
|  - IIFE 실행 시 Stack에 지역변수 생성              |
|  - IIFE 종료 후 Closure Scope가 Heap에 유지       |
|  - 반환된 함수들이 Closure를 참조하므로 GC 안됨    |
|  - 외부에서 _key, _value 직접 접근 불가            |
+--------------------------------------------------+
```

### Stack vs Heap 생명주기 비교

| 항목 | Stack | Heap |
|------|-------|------|
| 저장 대상 | 원시값, 참조 주소 | 객체, 함수, 클로저 |
| 생명주기 | 함수 종료 시 자동 해제 | GC가 관리 |
| 접근 속도 | 빠름 (100%) | 상대적 느림 (70~80%) |
| 크기 제한 | 제한적 (보통 1MB) | 상대적 큼 |
| 클로저 변수 | 실행 중에만 존재 | Closure Scope로 유지 |

## 왜 결합되었는가?

| 필요성 | 해결책 |
|--------|--------|
| 데이터 저장 | 프로퍼티 |
| 데이터 보호 | 클로저 |
| 동적 생성 | 함수 |
| 행위 정의 | 메서드 (함수 프로퍼티) |

**JavaScript는 이 모든 것을 하나의 언어에서 지원**하기 위해 결합했다.

## JavaScript의 설계 철학

### 영향을 받은 언어

```
+--------------------------------------------------+
|              JavaScript의 뿌리                    |
+--------------------------------------------------+
|                                                   |
|  Scheme (함수형 언어)                             |
|  +-- 일급 함수 (First-class Function)            |
|  +-- 클로저 (Closure)                            |
|  +-- 람다 표현식                                  |
|                                                   |
|  Self (프로토타입 기반 OOP)                       |
|  +-- 프로토타입 상속                              |
|  +-- 객체 리터럴                                  |
|  +-- 동적 프로퍼티                                |
|                                                   |
+--------------------------------------------------+
```

### 결합의 결과

```
+--------------------------------------------------+
|              JavaScript의 특징                    |
+--------------------------------------------------+
|                                                   |
|  함수 = 객체                                      |
|  +-- 함수도 프로퍼티를 가질 수 있음                |
|  +-- 함수를 변수에 할당 가능                       |
|  +-- 함수를 인자로 전달 가능                       |
|  +-- 함수를 반환값으로 사용 가능                   |
|                                                   |
+--------------------------------------------------+
```

## 실전 비교: element.js 분석

### 현재 코드 (객체 리터럴)

```js
// js/es5/section2-3/object/literal/element.js
var element = {
    key: "",
    value: ""
};
```

**특징:**
- 단순한 데이터 구조
- 외부에서 직접 접근/수정 가능
- 보호 메커니즘 없음

### 클로저 적용 시

```js
var element = (function() {
    var _key = "";
    var _value = "";

    return {
        getKey: function() { return _key; },
        setKey: function(k) { _key = k; },
        getValue: function() { return _value; },
        setValue: function(v) { _value = v; }
    };
})();
```

**특징:**
- 캡슐화된 데이터 구조
- getter/setter로만 접근 가능
- 비공개 상태 보호

---

### 캡슐화(Encapsulation) 개념

```
+--------------------------------------------------+
|              캡슐화란?                             |
+--------------------------------------------------+
|                                                   |
|  정의: 데이터와 그 데이터를 조작하는 메서드를        |
|        하나의 단위로 묶고, 외부 접근을 제한하는 것   |
|                                                   |
|  목적:                                            |
|  +-- 정보 은닉 (Information Hiding)              |
|  +-- 데이터 무결성 보장                           |
|  +-- 인터페이스와 구현 분리                        |
|  +-- 유지보수성 향상                              |
|                                                   |
+--------------------------------------------------+
```

### OOP 캡슐 vs 클로저 캡슐 비교

| 항목 | OOP 캡슐 (Class 기반) | 클로저 캡슐 (ES5) |
|------|----------------------|------------------|
| 언어 | Java, C++, ES6 Class | JavaScript ES5 |
| 키워드 | `private`, `public` | 없음 (스코프 활용) |
| 접근 제어 | 명시적 (키워드) | 암묵적 (클로저) |
| 상속 지원 | 네이티브 지원 | 프로토타입 체인 |
| 메모리 효율 | 높음 (프로토타입 공유) | 낮음 (인스턴스마다 클로저) |
| 구현 난이도 | 쉬움 (30%) | 보통 (50%) |
| 유연성 | 보통 (60%) | 높음 (85%) |
| 런타임 보호 | 강함 (95%) | 완벽 (100%) |

### 캡슐화 강도 비교 (퍼센트)

```
+--------------------------------------------------+
|              캡슐화 강도 비교                       |
+--------------------------------------------------+
|                                                   |
|  [OOP 캡슐 - private 키워드]                      |
|  +-- 컴파일 타임 체크: 100%                       |
|  +-- 런타임 보호: 95% (리플렉션 우회 가능)         |
|  +-- 상속 시 보호: 90%                            |
|  +-- 전체 캡슐화 강도: 약 95%                     |
|                                                   |
|  [클로저 캡슐 - 스코프 활용]                       |
|  +-- 컴파일 타임 체크: 0% (동적 언어)              |
|  +-- 런타임 보호: 100% (스코프 외부 접근 불가)     |
|  +-- 상속 시 보호: 100% (클로저 독립)             |
|  +-- 전체 캡슐화 강도: 약 100%                    |
|                                                   |
+--------------------------------------------------+
```

### 메모리 효율성 비교

| 인스턴스 수 | OOP (메서드 공유) | 클로저 (메서드 복제) |
|------------|------------------|---------------------|
| 1개 | 100% | 100% |
| 10개 | 100% | 150% |
| 100개 | 100% | 200% |
| 1000개 | 100% | 300%+ |

**설명:**
- OOP: 메서드가 프로토타입에 1번만 저장
- 클로저: 인스턴스마다 함수 객체 + 클로저 스코프 생성

### 언제 무엇을 선택하는가?

| 상황 | 권장 방식 | 이유 |
|------|----------|------|
| 많은 인스턴스 생성 | OOP (프로토타입) | 메모리 효율 (100%) |
| 완벽한 은닉 필요 | 클로저 | 런타임 보호 (100%) |
| ES5 환경 | 클로저 | Class 미지원 |
| ES6+ 환경 | Class + # | 가독성 + 은닉 |
| 싱글톤/모듈 | 클로저 (IIFE) | 1회성이므로 효율적 |

## 결론

```
+--------------------------------------------------+
|              정리                                  |
+--------------------------------------------------+
|  프로퍼티 = 데이터 (상태)                          |
|  함수 = 행위 + 동적 생성                          |
|  클로저 = 함수 + 스코프 기억 (캡슐화)              |
|                                                   |
|  역사: 함수 -> 클로저 -> 프로퍼티(OOP)            |
|  JS: 모두 결합하여 유연성 극대화                   |
+--------------------------------------------------+
```

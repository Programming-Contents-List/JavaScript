# LRU 알고리즘을 구현하면서

> 해당 알고리즘을 구현하면서 본인은 반복문에만 집중해서 문제를 풀어내려고만 생각했다. 하지만, 실제 구현에서는 데이터를 중심으로 구현을 해야했다. 즉, 어떻게 데이터를 이해하고 활용하는지를 기준으로 작성해야 했던 것이다.

---

## 피드백

### 문제 인식

| 기존 사고 방식 | 실제 필요한 사고 방식 |
|----------------|----------------------|
| "어떻게 반복할까?" (Loop 중심) | "무엇을 다룰까?" (데이터 중심) |
| 하나의 함수에서 전부 처리 | 역할별 함수 분리 |
| 알고리즘 = 반복문 | 알고리즘 = 자료구조 + 연산 |

### 구체적 문제점

1. **Loop 먼저 생각함**
   - for문을 어떻게 돌릴지 먼저 고민
   - 데이터 구조는 나중에 생각

2. **함수 분리 사고 부족**
   - `removeFromOrder`를 별도 함수로 분리해야 한다는 생각을 못 함
   - 모든 로직을 하나의 함수에서 처리하려 함

3. **자료구조 선택 기준 부재**
   - cache가 왜 Object인지
   - order가 왜 Array인지 명확히 인식하지 못함

## 개선 방안

### 알고리즘 접근 순서 변경

```
[기존]
1. 반복문 구상
2. 조건문 추가
3. 데이터 끼워 맞추기

[개선]
1. 입력/출력 데이터 정의
2. 필요한 자료구조 선택 (왜 이 구조인가?)
3. 자료구조에 필요한 연산 정의
4. 연산 내부에서 loop 필요 시 작성
```

### 체크리스트 (알고리즘 문제 풀 때)

- [ ] 어떤 데이터를 저장해야 하는가?
- [ ] 어떤 자료구조가 적합한가? (Array, Object, Map, Set...)
- [ ] 그 자료구조를 선택한 이유는?
- [ ] 필요한 연산(함수)은 무엇인가?
- [ ] 각 연산은 독립적으로 분리 가능한가?
- [ ] 반복문은 어느 연산 내부에서 필요한가?

### 실천 과제

1. **의사코드 먼저 작성하기**
   - 코드 작성 전 의사코드로 흐름 정리
   - 데이터 → 연산 → 세부 구현 순서로

2. **자료구조 공부**
   - 각 자료구조의 특징과 시간복잡도
   - 언제 어떤 자료구조를 선택하는지

3. **함수 분리 연습**
   - 하나의 함수 = 하나의 책임 (SRP)
   - 재사용 가능한 단위로 분리

## 오늘의 깨달음

> **알고리즘 = 반복문**이 아니라 **알고리즘 = 자료구조 + 연산**이다.

```
코딩 테스트 사고         vs        실무/모듈화 사고
─────────────────────────────────────────────────
하나의 함수에서 전부              역할별로 분리

function LRU() {                 function get() { ... }
    for (...) { ... }            function set() { ... }
    if (...) { ... }             function removeFromOrder() { ... }
}
```
